# UI v2 — breathing, overflow primitives, D+A hybrid theme

> **Status:** design approved by user through three iterative HTML previews (`/tmp/datalearn-ui-breathing-preview.html`, `/tmp/datalearn-theme-demos.html`, `/tmp/datalearn-theme-hybrid.html`). Awaits Codex adversarial review and writing-plans handoff.
> **Date:** 2026-05-23
> **Builds on:** `docs/design-system/` (the existing visual contract — extended, not replaced).

## Goal

Three coupled improvements that share the same migration cadence and the same target files:

1. **Container breathing.** Pages no longer leave 144–384px gutters on common screen sizes (1440–1920px). A new `Container width="2xl"` (max-w-screen-2xl = 1536px) absorbs the dashboards (catalog, learn index, admin lists); the article reader narrows from `xl` to `md` for proper reading line length.

2. **Overflow primitives.** Wide tables, code blocks, and Mermaid diagrams that exceed the container scroll *cleanly* with a right-edge fade gradient that signals "more content here" and disappears at scroll-end. Code blocks gain a soft-wrap toggle that survives per-user.

3. **Theme refresh — D+A hybrid.** Engineer-tool polish (Linear / Vercel / Railway visual register) as the base, with terminal flavor (`[bracket]` eyebrows, monospace stat labels, `$` prompts, `▸` status arrows, `⌘↵` keyboard hints) applied **only on code-adjacent surfaces** (workspace, admin tables, result panels, problem catalog stat blocks). Prose, marketing, forms, and navigation stay clean D. **Dark becomes the new default**; light is retained as a refined toggle.

All three ship together in a single PR cycle because they touch the same files (`app/globals.css`, `components/ui/Container.tsx`, `components/sql/ResultTable.tsx`, the markdown renderer, every page's `Container` call), share the same migration sequencing, and reinforce each other (the new theme depends on the new primitives; the primitives use the new tokens).

UI v2 also **adopts `shadcn/ui`** for the a11y-heavy primitives that are genuinely hard to hand-roll well (Dialog, Sheet, Popover, Tooltip, Toast, Command, DropdownMenu, Tabs, Form, Separator, ScrollArea). Existing hand-rolled primitives stay. shadcn's default CSS variables match the existing token names by convention, so adoption is additive — no value migration required.

## Scope expansion (2026-05-23 mid-spec)

After Codex pass 5, the user expanded scope to cover **every page** including the admin portal — what was previously "UI v3 follow-up" is now part of UI v2. This means UI v2 includes:

- Public surfaces: home, practice catalog + workspace, learn lanes + topics + articles, tracks, daily, profile.
- Admin surfaces: dashboard, problems list/new/edit, articles list/new/edit/review, tracks list/new/edit, discussions + moderation, schemas, assets, users/moderators.
- Legal: terms, privacy.
- Auth surfaces: sign-in, sign-out flows.

Concretely the migration table below grows from ~10 surfaces to ~25. The migration-order section grows from 6 commits to ~10. Total effort estimate roughly doubles vs. the "v2 + defer admin to v3" baseline.

The user accepted this trade because:
- The shared primitives (tokens, Container, shadcn Dialog/Popover/etc.) land once and benefit every surface.
- Separating admin into a v3 PR would mean re-touching every shared file weeks later for no architectural gain.
- The admin dashboard redesign benefits from being designed alongside the rest of the new theme (consistent feel, shared chrome).

## Non-goals

- New navigation IA (`Navbar` structure stays as-is; only its tokens shift).
- A typography overhaul. Inter + JetBrains Mono are retained. Only headline weight + tracking shift slightly.
- Marketing copy or messaging changes.
- New animation system. Existing 150/200ms `transition-colors`/`transition-[border,shadow,translate]` pattern is kept; no new motion primitives.
- Replacing `next-themes`. Theme toggle stays under the existing hook.
- Replacing the Prism syntax-highlighter theme. Out of scope; tracked as a separate cleanup if it conflicts with the new token palette.
- Mobile-app-specific UI work. Site is responsive web only.

## Token contract

### Compatibility invariant (critical — do not violate)

**Every CSS variable that exists in `app/globals.css` today MUST continue to exist with the same name after UI v2.** This includes the full existing set (verified against the current `:root` and `.dark` blocks):

`--background`, `--foreground`, `--surface`, `--surface-muted`, `--surface-elevated`, `--border`, `--border-strong`, `--muted`, `--muted-foreground`, `--primary`, `--primary-foreground`, `--primary-hover`, `--accent`, `--accent-foreground`, `--destructive`, `--destructive-foreground`, `--warning`, `--warning-foreground`, `--success`, `--success-foreground`, `--easy`, `--easy-bg`, `--easy-fg`, `--medium`, `--medium-bg`, `--medium-fg`, `--hard`, `--hard-bg`, `--hard-fg`, `--ring`, `--radius`.

These power the existing Tailwind utilities (`bg-background`, `text-foreground`, `bg-surface-muted`, `bg-surface-elevated`, `hover:bg-primary-hover`, `text-destructive`, `bg-easy-bg`, `text-easy-fg`, `border-border`, `border-border-strong`, etc.) that hundreds of components rely on. **UI v2 changes the *values* of these tokens for the new dark default and the new (inverted) light mode; it does not rename any token.**

The few net-new utilities UI v2 introduces (see "New utility classes" below) read existing tokens via `hsl(var(--<existing-name>))`. No new CSS variables are required except where called out explicitly under "Net-new tokens" — and each addition is documented.

### Class-on-html contract (critical — Tailwind `dark:` variants depend on this)

The existing codebase uses Tailwind `dark:` variants in six files (`MarkdownRenderer.tsx`, `ProblemPanel.tsx`, the legal pages, the tracks page, others). Those variants fire ONLY when `<html class="dark">` is present. If UI v2 moved tokens to `:root` and stopped adding the `.dark` class, every `dark:prose-invert` and similar utility would silently no-op — Learn article prose would render light typography on dark token surfaces.

The token structure therefore is:

```css
:root {
    /* dark values — SSR fallback before hydration or for users with no theme preference */
}
.dark {
    /* dark values — explicitly redundant with :root. Required so Tailwind's
       dark: variants fire when next-themes adds class="dark" to <html>. */
}
.light {
    /* light values — applied when next-themes adds class="light" to <html> */
}
```

`next-themes` configuration:
- `defaultTheme: "dark"`
- `attribute: "class"` (already the default)
- `themes: ["dark", "light"]`
- `enableSystem: true` is preserved if currently enabled; user system preference still wins on first load.

This means new users land with `<html class="dark">` (or `class="light"` if their OS prefers light and `enableSystem` honors that). The Tailwind `dark:` variants continue to work unchanged. The `:root` block exists only as an SSR-flash safety net.

**Done criteria adds a grep gate**: every `dark:`-prefixed Tailwind utility in the current tree must continue to render correctly under both `class="dark"` and `class="light"`. Concretely, after migration, run `rg "dark:" components app` and visually verify each of the ~6 sites renders correctly in light mode (toggle ON) — because the `dark:` style is suppressed there and the underlying base style must look right.

### Net-new tokens (additive only)

Only one CSS variable is genuinely new — the rest of UI v2 reuses existing tokens or uses opacity:

```css
:root {
    --muted-foreground-dim: 220 9% 42%;   /* dimmer than --muted-foreground */
}
.light {
    --muted-foreground-dim: 220 9% 64%;
}
```

**Decoration-only contract — read this carefully.** `--muted-foreground-dim` is below the WCAG AA contrast threshold (~2.6:1 on white, ~2.9:1 on dark surface-muted) and **must not be used for content-bearing text**. Approved uses:

- `.bracket::before` / `.bracket::after` rendering the `[` and `]` characters around an eyebrow. The bracket characters carry no information; the label text inside uses `--primary` (AA-passing).
- Pure decorative dot separators in stat-block labels (e.g., `<span class="text-muted-foreground-dim">·</span>` between metadata fragments).

Any author tempted to use it for actual label content must instead use `--muted-foreground` (AA-passing at 11px against `--surface` in both modes) or the new tuned-down `--easy/medium/hard` values.

### shadcn token compatibility map

shadcn primitives reference seven CSS variables that the current `app/globals.css` does NOT define. They MUST be added to the token block at the same time the existing token values are re-toned. Each is aliased to an existing UI v2 token so the values automatically follow mode changes — no separate dark/light tuning required.

Add these lines to `:root`, `.dark`, and `.light` blocks alike (the `var(--…)` reference resolves to the active mode's value of the target token):

```css
:root, .dark, .light {
    /* shadcn surface family — aliased to UI v2 surfaces */
    --card: var(--surface);
    --card-foreground: var(--foreground);
    --popover: var(--surface-elevated);     /* popovers float; map to elevated */
    --popover-foreground: var(--foreground);

    /* shadcn secondary button family — aliased to muted surface */
    --secondary: var(--surface-muted);
    --secondary-foreground: var(--foreground);

    /* shadcn input border — aliased to the standard border token */
    --input: var(--border);
}
```

Plus the Tailwind v4 `@theme inline` mappings (so shadcn's `bg-card`, `text-popover-foreground`, etc. resolve to utility classes). Add to the existing `@theme inline` block:

```css
@theme inline {
    /* existing mappings unchanged */
    --color-card: hsl(var(--card));
    --color-card-foreground: hsl(var(--card-foreground));
    --color-popover: hsl(var(--popover));
    --color-popover-foreground: hsl(var(--popover-foreground));
    --color-secondary: hsl(var(--secondary));
    --color-secondary-foreground: hsl(var(--secondary-foreground));
    --color-input: hsl(var(--input));
}
```

**Why aliases rather than new HSL values:** the seven additions are *secondary semantic roles* that, in our visual system, are already covered by `--surface`, `--surface-elevated`, `--surface-muted`, and `--border`. Aliasing means there's a single source of truth for each visual role — when we tune `--surface`, cards follow automatically. The shadcn skill's customization guide explicitly endorses this aliasing pattern.

**Not added in UI v2** (defer to when actually needed):

- `--chart-1` through `--chart-5` — shadcn charts; no charts in v2. Add when UI v3 brings dashboard charts.
- `--sidebar-*` family — shadcn Sidebar component; not used in v2. Likely added by UI v3 admin redesign.

**Verification step in implementation** (added to Done criteria): after `npx shadcn@latest add dialog tooltip sonner`, run

```bash
rg -hoE 'var\(--[a-z0-9-]+\)' components/ui | sort -u | while read v; do
    name=$(echo "$v" | sed -E 's/var\((--[a-z0-9-]+)\)/\1/')
    grep -q -- "$name" app/globals.css || echo "MISSING: $name"
done
```

This grep extracts every `var(--…)` reference from `components/ui/` and confirms each one is defined in `app/globals.css`. Implementation MUST exit with no `MISSING:` lines before commit.

### Contrast verification

Every token-on-surface pair used by the new UI v2 surfaces must pass WCAG 2.1 AA at the relevant text size:

| Token | On | Required ratio | Notes |
|---|---|---|---|
| `--foreground` | `--background`, `--surface`, `--surface-muted`, `--surface-elevated` | 4.5:1 | All four surface variants. |
| `--muted-foreground` | `--surface`, `--surface-muted` | 4.5:1 | Used at 11px in stat labels. |
| `--primary` | `--background`, `--surface` | 4.5:1 | Text + outlined-button label. Adjusted in light mode (see above). |
| **`--primary-foreground`** | **`--primary`** | **4.5:1** | **Filled primary-button text. Must be verified in BOTH modes.** |
| `--accent` | `--background`, `--surface` | 4.5:1 | Hint text and "[ note ]" labels. |
| **`--accent-foreground`** | **`--accent`** | **4.5:1** | **Filled accent-button text. Pass-2 found the original near-white-on-light-violet pair at 2.23:1 — fixed to a dark-on-light-violet pair in dark mode.** |
| `--destructive` | `--background`, `--surface` | 4.5:1 | Error and reject states. |
| **`--destructive-foreground`** | **`--destructive`** | **4.5:1** | **Filled destructive-button text (e.g., "Delete" confirmations).** |
| **`--warning-foreground`** | **`--warning`** | **4.5:1** | **Filled warning surfaces (toasts, pills inside `--warning-bg` contexts).** |
| **`--success-foreground`** | **`--success`** | **4.5:1** | **Filled success surfaces.** |
| `--easy` / `--medium` / `--hard` | `--surface` | 4.5:1 | At 11px in difficulty badges. Light-mode values tuned darker. |
| `--easy-fg` / `--medium-fg` / `--hard-fg` | `--easy-bg` / `--medium-bg` / `--hard-bg` | 4.5:1 | Inside pill backgrounds. |
| `--ring` | `--background` | 3:1 | Focus indicator, large-area shape. |
| `--muted-foreground-dim` | any | **N/A — decoration only** | Per the contract above. |

**Rule:** any token named `--*-foreground` must pair-pass against its base counterpart in both modes. Codex pass 3 caught the `--accent-foreground` violation that the original matrix missed.

**Implementation must run a real contrast calculator** (Lighthouse, axe-core, or `@adobe/leonardo-contrast-colors`) against every pair above and confirm it passes before merge. The HSL values in the spec are starting points; if any combination measures below the threshold on a real Lighthouse run, the value is adjusted (not the contract).

The shift from v1's "green primary on light surface" to "teal primary on dark surface" is the headline visual change. v1 green is preserved as the `--easy` token (with new -bg/-fg pair tuned for dark), so submit-accepted and easy-difficulty indicators feel familiar.

## New React primitives

All in `components/ui/`. Each is small, focused, fully typed, semantic-token-only.

### `<Eyebrow>` / `<BracketLabel>`

```tsx
// components/ui/Eyebrow.tsx
interface EyebrowProps {
    children: React.ReactNode
    variant?: "plain" | "bracket"
    className?: string
}
export function Eyebrow({ children, variant = "plain", className }: EyebrowProps) {
    return (
        <div className={cn(
            "text-[11px] font-mono uppercase tracking-widest",
            variant === "bracket" ? "bracket" : "text-muted-foreground",
            className
        )}>{children}</div>
    )
}
```

Use: above page titles (`<Eyebrow variant="bracket">CATALOG</Eyebrow>`), as labels on stat blocks, as section delimiters in the workspace chrome.

### `<StatusPill>`

```tsx
// components/ui/StatusPill.tsx
type Status = "accepted" | "rejected" | "pending" | "draft"
interface StatusPillProps {
    status: Status
    label?: string  // overrides default capitalized status
    icon?: React.ReactNode  // optional leading icon
    className?: string
}
const DEFAULT_LABEL: Record<Status, string> = {
    accepted: "▸ accepted",
    rejected: "✗ rejected",
    pending: "⊗ pending",
    draft: "· draft",
}
export function StatusPill({ status, label, icon, className }: StatusPillProps) {
    return (
        <span className={cn("pill", `pill-${status}`, className)}>
            {icon}
            <span>{label ?? DEFAULT_LABEL[status]}</span>
        </span>
    )
}
```

Use: result panel status (`<StatusPill status="accepted" />`), submission history badges, draft article markers, asset status in admin.

### `<ScrollableTable>`

```tsx
}
export function ScrollableTable({ children, className }: ScrollableTableProps) {
    const ref = useRef<HTMLDivElement>(null)
    const [atEnd, setAtEnd] = useState(true) // start at-end (no fade) until measured

    useEffect(() => {
        const el = ref.current
        if (!el) return
        const check = () => {
            const reachedEnd =
                Math.abs(el.scrollLeft + el.clientWidth - el.scrollWidth) < 4
            const overflows = el.scrollWidth > el.clientWidth + 1
            setAtEnd(!overflows || reachedEnd)
        }
        check()
        el.addEventListener("scroll", check, { passive: true })
        const observer = new ResizeObserver(check)
        observer.observe(el)
        return () => {
            el.removeEventListener("scroll", check)
            observer.disconnect()
        }
    }, [])

    return (
        <div
            ref={ref}
            className={cn(
                "scroll-fade overflow-x-auto scrollbar-thin",
                atEnd && "at-end",
                className
            )}
        >
            {children}
        </div>
    )
}
```

Use: wrap any wide content (ResultTable, admin tables, prose-table inside MarkdownRenderer, Mermaid SVG container).

## shadcn/ui adoption

UI v2 introduces `shadcn/ui` as the source for a11y-heavy primitives. The existing hand-rolled primitives stay; shadcn is **additive**, not a replacement.

### Why add shadcn now

Three concrete v2 needs require primitives that hand-rolling well is genuinely hard:

- **Sign-in modal** (already specified in the anonymous-gating spec) — needs a proper `<Dialog>` with focus trap, escape-to-close, scroll lock, and SSR-safe portal.
- **Command palette** (`⌘K` hint surfaced in the new chrome) — `<Command>` from shadcn (built on `cmdk`) handles fuzzy search, keyboard nav, and group semantics.
- **Status toasts** for things like "Free run used", copy-paste confirmations, and admin success messages — `sonner` (the toast library shadcn ships) is the industry-standard answer.

Three more that we'll want soon (UI v3 admin redesign, or as Learn pages grow):

- `<Tooltip>` for keyboard-shortcut explainers on icon buttons.
- `<Popover>` for the difficulty/tag filter dropdowns on `/practice`.
- `<Tabs>` (shadcn's, with proper `TabsList` semantics) to replace the inline tab logic in `PracticeList` and `ProblemPanel`.

### Initialization

The project context (`npx shadcn@latest info`) confirms: Next.js 16 App Router, RSC, TypeScript, Tailwind v4, `@/` import alias, `app/globals.css` as the global stylesheet, `components.json` does not yet exist. `init` is therefore a fresh setup.

### Coexistence with hand-rolled primitives

shadcn's `Button` and our existing `Button` (`components/ui/Button.tsx`) **both stay**. We do not migrate the existing one because:

- Every page imports `@/components/ui/Button` already. A global rename is a large diff with no visual benefit.
- The hand-rolled `Button` has the right variants and the right tokens.
- shadcn's `Button` is available for new code if a contributor prefers it — both render identically because they consume the same tokens.

Same logic for `Card`, `Badge`, `Input`. If a v2.1 or v3 cleanup wants to standardize on shadcn for these too, it's a follow-up — out of v2 scope.

### Rules from the shadcn skill that this spec inherits

The shadcn skill defines critical rules ([rules/styling.md](file:rules/styling.md), [rules/composition.md](file:rules/composition.md), etc.). UI v2 implementation must follow them:

- No `space-x-*` / `space-y-*`; use `flex` with `gap-*`.
- `size-*` for square dimensions, not `w-*` `h-*`.
- No `truncate` longhand.
- Components handle their own `z-index`; no manual overrides.
- `Dialog` / `Sheet` need a `DialogTitle` / `SheetTitle` (use `className="sr-only"` if visually hidden).
- Form fields use `FieldGroup` + `Field` composition (relevant when forms are migrated in v3).
- Icons in `Button` use `data-icon="inline-start"` / `data-icon="inline-end"`, no sizing classes.
- `Button` has no `isPending`/`isLoading` prop — compose with `Spinner` + `disabled`.

These rules apply only to **new** code introduced in UI v2 or to shadcn components. Existing hand-rolled primitives don't have to retrofit immediately.

## Reduced-motion + accessibility

- `prefers-reduced-motion: reduce` already disables `.cursor` animation (specified above) and existing project transitions. The new `scroll-fade` is a static gradient — no motion concerns.
- Status pills carry both icon + text so color is never the sole channel of meaning.
- `<Kbd>` renders as semantic `<kbd>` element; screen readers announce keyboard shortcuts naturally.
- All new tokens maintain WCAG AA contrast against the surfaces they sit on. Light-mode inverted set re-verified at spec implementation.
- The default-dark switch must keep light mode equally accessible — light is not deprecated, just deprioritized. WCAG AA passes on both modes for the `--primary` on `--surface` pair.

Post-init verification step confirms `app/globals.css` is unchanged from the v2 token commit. Done criterion mandates this check. |
| **shadcn primitives reference tokens our globals.css doesn't define** (`--card`, `--popover`, `--secondary`, `--input` and their `-foreground` pairs). Without these, Dialog/Tooltip/Sonner ship with invalid backgrounds and unreadable text. Caught by Codex pass 5. | New "shadcn token compatibility map" subsection enumerates all seven missing variables and aliases each to an existing UI v2 token via `var(--…)` (so mode changes propagate automatically). Done criterion adds a grep verification step that confirms every `var(--…)` reference in `components/ui/` resolves to a defined token in `app/globals.css`. |
| **shadcn-added components clash visually with hand-rolled siblings.** A new shadcn `Button` would look subtly different from `components/ui/Button.tsx` if it consumed a different token set. | shadcn's defaults read from `--background`, `--foreground`, `--primary`, etc. — the same token names we already use. Visual parity comes for free if our token values are correct. The Done criterion's contrast-matrix verification covers this. Pages introducing a shadcn primitive next to a hand-rolled one (e.g. Dialog inside a Card) are screenshot-checked in both modes. |
| **A new contributor adds a hand-rolled `<div>` that should be a shadcn primitive (e.g. raw modal markup with manual focus trap).** Hard to catch in review without a culture; over time the hand-rolled side drifts back. | Spec includes the shadcn skill's "Use Components, Not Custom Markup" rule list. PR template gets a checkbox: "If this PR adds an overlay, dropdown, tooltip, or toast, it uses a shadcn primitive." Not enforced by CI; relies on review discipline. |
| Code-block soft-wrap toggle confuses learners | Default is scroll (current behavior). Toggle is a small icon button top-right with a tooltip "Wrap long lines (W)". localStorage-persisted; per-user, per-codeblock-class. Not a critical-path UI element. |
| `next-themes` SSR flash | Already mitigated by the existing nonce + ThemeProvider integration. The default flip doesn't change SSR semantics — only the initial token set. |
| Migration breaks the curriculum article authoring (curriculum content is mid-flight) | Article content uses the markdown renderer, which is migrated as part of step 1 (tokens) + step 4 (article reader narrowing). Author prompt is content-only, unaffected. Verified during step 4. |
